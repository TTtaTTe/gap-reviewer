# Deliverable Gap Reviewer

## 목적

지시서(요청서)의 **주요 요구사항**이 작업물에 빠짐없이 반영되었는지 확인하는 멀티에이전트 리뷰 시스템.

## 핵심 원칙 (모든 에이전트에 반드시 적용)

1. **큰 항목 누락 검출이 목적** - 단어의 어감, 표현 방식, 문체 등은 검토 대상이 아님
2. **꼬투리 잡기 금지** - "~합니다 vs ~한다" 같은 문체 차이, 어감 차이는 무시
3. **주요 요구사항 중심** - 지시서가 명시적으로 요청한 핵심 항목만 체크
4. **실용적 보완 제안** - 문제 지적에 그치지 않고, 방향 + 작성 예시를 함께 제시

## 출력 양식 원칙 (모든 Phase 공통)

**반드시 아래 정해진 양식만 출력한다.** 추가 테이블, 요약, 부연설명, 해설을 붙이지 않는다.
각 Phase/Step의 "출력 양식" 섹션에 정의된 형식을 그대로 따른다.

## 입력

- 지원 형식: PDF, DOCX, PPTX, XLSX
- 인자: `$ARGUMENTS` 로 두 파일 경로를 받거나, 대화에서 파일을 요청
  - `$ARGUMENTS[0]`: 지시서(요청서) 파일 경로
  - `$ARGUMENTS[1]`: 작업물 파일 경로 (없으면 Phase 1 완료 후 요청)

## 채점 기준

상세 채점 기준은 [scoring_rubric.md](references/scoring_rubric.md) 참조.

## JSON 출력 스키마 및 저장 규칙

gap-fix 스킬과의 데이터 연결을 위해, Phase 2 완료 후 반드시 JSON 파일을 생성한다.
상세 스키마는 [output_schema.md](references/output_schema.md) 참조.

### 저장 경로 우선순위
1. 지시서 파일과 **같은 폴더**
2. 쓰기 불가 시 → 사용자 **Downloads 폴더**
3. 그래도 불가 시 → 사용자에게 경로 질문

### 파일명 형식
`검토보고서_YYYYMMDD_HHMMSS.json` (MD, DOCX와 동일 타임스탬프)

---

## 워크플로우 실행

### 파일 읽기

1. `$ARGUMENTS[0]`이 있으면 Read 도구로 지시서 파일을 읽는다. 없으면 사용자에게 요청한다.
2. 읽은 내용을 이후 에이전트들에게 전달한다.

## Phase 1: 지시서 분석 (에이전트 3개, 순차)

### Step 1: 분석가 (Analyst) - 요구사항 추출

Agent 도구로 서브에이전트(subagent_type: general-purpose)를 생성한다.

프롬프트에 반드시 포함할 내용:
- 역할: "요구사항 분석가"
- 지시서 원문 전체
- 추출 기준: 주요 요구항목, 체크포인트, 요청사항을 항목별로 추출
- 원칙: 큰 항목 위주 추출, 세부 표현/어감/문체 무시
- 분류: "명시적 요구사항" / "암묵적 요구사항" / "기타 조건/제약사항"

출력 양식 (이 형식만 출력, 다른 형식 금지):
```
## 요구사항 추출 결과

| # | 분류 | 요구항목 | 상세내용 | 근거위치 |
|---|------|---------|---------|---------|
| 1 | 명시적 | 항목명 | 상세 | 문서 N페이지 |
| ... | ... | ... | ... | ... |

총 추출 항목: N개
```

분석가의 출력을 기억해둔다.

### Step 2: 리뷰어 (Reviewer) - 검토 및 수정

Agent 도구로 서브에이전트를 생성한다.

프롬프트에 반드시 포함할 내용:
- 역할: "요구사항 리뷰어" - 분석가의 결과를 검토하고, 누락/오류를 **직접 수정**한다
- 지시서 원문 전체 + 분석가 결과
- 검토 기준: 완전성(2.5) + 정확성(2.5) + 구체성(2.5) + 분류체계(2.5) = 10.0점
- 원칙: 꼬투리 잡기 금지, 주요 항목 누락만 지적
- 작업:
  1. 분석가가 놓친 항목이 있으면 **직접 추가** (피드백만 주지 말고 직접 수정)
  2. 잘못 해석된 항목이 있으면 **직접 수정**
  3. 추가/수정된 항목에 [보완됨] 표시
  4. 신뢰도 점수 채점 (10.0 기준)

출력 양식 (이 형식만 출력):
```
## 리뷰 완료 - 수정된 요구사항 목록

| # | 분류 | 요구항목 | 상세내용 | 근거위치 | 비고 |
|---|------|---------|---------|---------|------|
| 1 | 명시적 | 항목명 | 상세 | N페이지 | |
| 2 | 명시적 | 항목명 | 상세 | N페이지 | [보완됨] |
| ... | ... | ... | ... | ... | ... |

수정 사항: N개 추가, N개 수정, N개 삭제
신뢰도 점수: X.X / 10.0
```

### Step 3: 정리자 (Organizer) - 체크리스트 JSON 생성

Agent 도구로 서브에이전트를 생성한다.

프롬프트에 반드시 포함할 내용:
- 역할: "체크리스트 정리자"
- 리뷰어의 수정된 요구사항 목록 + 신뢰도 점수
- 출력 형식: **반드시 유효한 JSON만 출력. JSON 외의 텍스트, 설명, 마크다운을 절대 포함하지 않는다.**

```json
{
  "checklist": [
    {
      "id": 1,
      "category": "분류명",
      "requirement": "요구항목 이름",
      "detail": "상세 내용 (검증 가능한 수준)",
      "source": "explicit 또는 implicit",
      "priority": "required 또는 recommended"
    }
  ],
  "total_items": 숫자,
  "phase1_score": 숫자
}
```

- 원칙: 각 항목이 Phase 2에서 "있다/없다"로 판별 가능해야 함, 우선순위 구분

### Phase 1 완료 알림

사용자에게 **반드시 아래 양식만** 출력한다. 추가 표, 카테고리 요약, 해설을 붙이지 않는다:

```
Phase 1 분석 완료
- 추출된 요구항목: {total_items}개 (필수: X개, 권장: Y개)
- 분석 신뢰도: {phase1_score}/10.0

작업물(보고서) 파일을 첨부해주세요.
```

`$ARGUMENTS[1]`이 있으면 바로 Phase 2로 진행한다.

## Phase 2: 작업물 대조 검토 (에이전트 5개, 4개 병렬 + 1개 순차)

Read 도구로 작업물 파일을 읽는다.

### Round 1 + Round 2 동시 실행 (4개 병렬)

**중요**: 아래 4개 에이전트를 Agent 도구로 **동시에(한 번의 응답에서 4개 Agent 호출)** 생성한다.
Round 2는 Round 1의 결과를 보지 않고 독립적으로 수행하므로, 4개를 동시에 실행할 수 있다.

#### 에이전트 A: 항목 대조자 - Round 1

Agent 도구로 서브에이전트를 생성한다.

프롬프트에 반드시 포함할 내용:
- 역할: "항목 대조자 (Round 1)" - 체크리스트 항목이 작업물에 존재하는지 1:1 대조
- 체크리스트 JSON + 작업물 전체 내용
- 채점: 10.0(완전충족) ~ 0.0(누락)
- 원칙: 표현 방식 달라도 내용 동일하면 충족, 어감/문체 차이 감점 안 함

출력 양식 (이 형식만 출력):
```
## 항목 대조 결과 (Round 1)

| # | 요구항목 | 점수 | 판단근거 |
|---|---------|------|---------|
| 1 | 항목명 | X.X | 근거 |
| ... | ... | ... | ... |
```

#### 에이전트 B: 품질 검토자 - Round 1

Agent 도구로 서브에이전트를 생성한다.

프롬프트에 반드시 포함할 내용:
- 역할: "품질 검토자 (Round 1)" - 충족 수준 심층 평가
- 체크리스트 JSON + 작업물 전체 내용
- 평가: 의도 반영 여부, 내용 깊이, 구성요소 포함 여부
- 원칙: 큰 그림 평가, 완벽한 문장 불요구
- 부족 항목에 "무엇이 부족한지" 구체 기술

출력 양식 (이 형식만 출력):
```
## 품질 검토 결과 (Round 1)

| # | 요구항목 | 점수 | 부족한 부분 |
|---|---------|------|-----------|
| 1 | 항목명 | X.X | 부족 내용 또는 "-" |
| ... | ... | ... | ... |
```

#### 에이전트 C: 항목 대조자 - Round 2

에이전트 A와 동일한 역할/프롬프트. 단, "Round 2"로 표기.
출력 양식도 동일 (제목만 "Round 2").

#### 에이전트 D: 품질 검토자 - Round 2

에이전트 B와 동일한 역할/프롬프트. 단, "Round 2"로 표기.
출력 양식도 동일 (제목만 "Round 2").

### 4개 에이전트 완료 후: 종합 보고자 (1개, 순차)

4개 에이전트의 결과가 모두 돌아온 후, Agent 도구로 종합 보고자 서브에이전트를 생성한다.

프롬프트에 반드시 포함할 내용:
- 역할: "종합 보고자"
- R1 항목대조 결과 + R1 품질검토 결과 + R2 항목대조 결과 + R2 품질검토 결과 + 체크리스트 JSON
- 점수 처리 규칙:
  - 각 항목: 4개 점수(R1대조, R1품질, R2대조, R2품질) 중 **최저점** 채택
  - 종합 점수 = 전체 항목 평균 (priority가 required인 항목은 1.5배 가중)
  - R1, R2 모두에서 10.0 미만 → [공통] 태그
  - R1 또는 R2 한쪽만 10.0 미만 → [참고] 태그
- 보완 제안: suggestion_direction(방향) + suggestion_example(작성 예시) 반드시 작성
- 출력: **반드시 유효한 JSON만 출력. JSON 외의 텍스트, 설명, 마크다운을 절대 포함하지 않는다.**

```json
{
  "schema_version": "1.0.0",
  "report": {
    "instruction_file": "파일명",
    "deliverable_file": "파일명",
    "instruction_file_path": "전체 경로",
    "deliverable_file_path": "전체 경로",
    "phase1_score": 숫자,
    "phase2_overall_score": 숫자,
    "total_items": 숫자,
    "fulfilled_count": 숫자,
    "partial_high_count": 숫자,
    "partial_count": 숫자,
    "insufficient_count": 숫자,
    "missing_count": 숫자
  },
  "items": [
    {
      "id": 1,
      "category": "분류명",
      "requirement": "요구항목",
      "detail": "상세 내용",
      "source": "explicit 또는 implicit",
      "priority": "required 또는 recommended",
      "score_r1": 숫자,
      "score_r2": 숫자,
      "final_score": 숫자,
      "status": "fulfilled/mostly_fulfilled/partial/insufficient/missing",
      "tag": "common/reference/null",
      "current_state": "현재 상태 또는 null",
      "suggestion_direction": "보완 방향 또는 null",
      "suggestion_example": "작성 예시 또는 null"
    }
  ],
  "phase1_history": [
    {"round": "분석가 추출", "score": null},
    {"round": "리뷰어 수정", "score": 숫자}
  ]
}
```

### Phase 2 완료 알림

사용자에게 **반드시 아래 양식만** 출력한다. 추가 해설, 요약 테이블을 붙이지 않는다:

```
Phase 2 대조 검토 완료

종합 점수: {phase2_overall_score} / 10.0
- 충족: {fulfilled_count}개
- 대부분충족: {partial_high_count}개
- 부분충족: {partial_count}개
- 미흡: {insufficient_count}개
- 누락: {missing_count}개

보고서를 생성합니다...
```

## 출력 생성

### 1. JSON 파일 생성

종합 보고자의 JSON 결과를 Write 도구로 저장한다.
파일명: `검토보고서_YYYYMMDD_HHMMSS.json`

### 2. MD 파일 생성

Write 도구로 마크다운 보고서를 생성한다.
파일명: `검토보고서_YYYYMMDD_HHMMSS.md`

MD 보고서 구조 (이 순서와 형식을 정확히 따른다):

```markdown
# 지시서-작업물 대조 검토 보고서

생성일시: YYYY-MM-DD HH:MM

## 1. 개요

| 항목 | 내용 |
|------|------|
| 지시서 파일 | {파일명} |
| 작업물 파일 | {파일명} |
| Phase 1 분석 신뢰도 | {점수}/10.0 |
| Phase 2 종합 점수 | {점수}/10.0 |

| 상태 | 항목 수 |
|------|---------|
| 충족 (10.0) | N |
| 대부분충족 (7.0~9.9) | N |
| 부분충족 (4.0~6.9) | N |
| 미흡 (1.0~3.9) | N |
| 누락 (0.0) | N |

## 2. 항목별 대조 결과

| # | 분류 | 요구항목 | R1 | R2 | 최종 | 상태 | 태그 |
|---|------|---------|----|----|------|------|------|
| 1 | 분류 | 항목명 | X.X | X.X | X.X | 상태 | 태그 |

## 3. 보완 필요 체크리스트

### 3-1. 누락 항목 (0.0점)

- [ ] **{요구항목}** ({점수}/10.0) [{태그}]
  - 현재 상태: {current_state}
  - 보완 방향: {suggestion_direction}
  - 작성 예시: "{suggestion_example}"

### 3-2. 미흡 항목 (1.0~3.9점)
(동일 형식)

### 3-3. 부분충족 항목 (4.0~6.9점)
(동일 형식)

### 3-4. 대부분충족 항목 (7.0~9.9점)
(동일 형식)

## 4. 충족 항목

| # | 분류 | 요구항목 | 최종점수 |
|---|------|---------|---------|

## 5. Phase 1 분석 이력

| 단계 | 점수 |
|------|------|
```

### 3. DOCX 파일 생성

종합 보고자의 JSON 결과를 임시 JSON 파일로 저장한 뒤, 아래 명령으로 DOCX를 생성한다:

```bash
python "${CLAUDE_SKILL_DIR}/scripts/generate_docx.py" "{json_path}" "{output_docx_path}"
```

파일명: `검토보고서_YYYYMMDD_HHMMSS.docx`
DOCX는 MD와 동일한 내용을 맑은고딕 폰트로 출력한다.

### 4. 완료 알림

**반드시 아래 양식만** 출력한다:

```
검토 완료!

종합 점수: {phase2_overall_score} / 10.0
- 충족: {fulfilled_count}개
- 보완 필요: {나머지}개

보고서 저장 위치:
- {json_path}
- {md_path}
- {docx_path}
```

## 오류 처리

1. 파일 읽기 실패 → "파일을 읽을 수 없습니다. 다른 형식으로 첨부해주세요."
